### 2. System Architecture and Pipeline
The architecture is modular:
trip generation -> simulation engine -> OR interface -> user choice -> state transition -> metrics.

RL is inserted only at OR-matched decision points.  
So RL does not replace OR planning; RL refines execution decisions at trip level.

Operationally, the pipeline is:
1. run OR and export plan,
2. translate to `u_odit.csv`,
3. generate trip stream in simulator,
4. run policy mode,
5. log transitions and KPI decomposition.

This allows apples-to-apples comparison under aligned seeds and time windows.

### 3. Scenario 1 Decision Logic
Scenario 1 action space is binary at each opportunity: offer vs no-offer.

User behavior is now single-layer:
- if offer exists: user chooses among offer, base, opt-out;
- if no offer: user chooses base or opt-out.

This is utility-based and sampled in real time.  
Scooter assignment remains deterministic highest-battery-first in Scenario 1.

### 4. Trip Generation
Trips are generated from aggregated OD profile `omega_od`.

Current arrival model is NB2 rather than Poisson, to reflect over-dispersion:
- mean from OD profile after scaling,
- dispersion from hourly phi profile.

This gives uncertainty in arrivals while keeping OR side unchanged.

### 5. Parameter Card
Let me highlight current settings:

Data/trip:
- 2-hour episodes, 15-minute slots,
- NB2 arrivals with by-hour phi,
- scaling currently: global 1.0, window 0 to 7, window scale 2.0, target OD scale 5.0.

RL:
- DDQN, hidden size 64,
- replay buffer 1000,
- epsilon decay from 1.0 to 0.05.

Evaluation:
- three-policy parallel protocol:
  always-offer, no-offer, and checkpoint,
  with aligned seeds and same demand draws.

### 6. Reward Logic
The reward is a hybrid:
- realized-loss term,
- delta EDL term,
- accept bonus,
- cost penalty,
- reject penalty.

Both realized-loss and delta EDL are normalized and clipped:
- realized-loss clipped to [0,1],
- delta EDL clipped to [-1,1].

Cost is counted on accept in current setup.

### 8. Current Three-Policy Results
For the current 400-episode evaluation:
- service rates are all very high and close,
- checkpoint uses fewer offers than always-offer,
- checkpoint obtains positive mean reward and positive mean delta EDL,
- always-offer has negative mean reward and negative delta EDL,
- no-offer is near zero baseline.

So the checkpoint policy is more selective and more efficient than always-offer under this setup.

### 9. Important Ongoing Point: Realized Loss
A key observation is that mean realized loss is currently zero across policies in this operating regime.

I have already updated the realized-loss accounting to an O + D1 + D2 joint scope in the decision window, based on feedback.  
Even with this revised scope, signal activation is still very limited under current supply-demand conditions.

So this is not a coding omission anymore; it is mainly a regime and sparsity issue.
